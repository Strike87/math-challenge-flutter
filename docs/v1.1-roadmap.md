## Roadmap Principle

v1.1 should feel like the same game with deeper ways to play. It should not look like a redesign. The original visual identity, avatar system, boss system, coin economy, achievements style, buttons, emoji language, confetti, sounds, and screen flow must stay familiar.

The Flutter advantage is not only prettier UI. The real advantage is structure: separate widgets, typed models, testable game rules, reusable services, and cleaner future expansion than the original one-file `index.html`.

## Release Gate Before Starting v1.1

Do not start v1.1 feature implementation until these are true:

| Gate | Requirement |
| --- | --- |
| Clone parity | Flutter v1.0 matches the original game behavior and core UI. |
| Release services | AdMob, IAP, restore purchases, and remove-ads entitlement are working. |
| Stability | `flutter analyze`, tests, debug build, and release build pass. |
| Device QA | Phone, tablet, and Android 15+ safe-area checks pass. |
| User feedback | v1.0 has been live long enough to catch obvious release bugs. |

## v1.1 Core Feature: Multi-Type Question System

This is the main v1.1 feature from the uploaded plan.

### Formats

| Format | Behavior | Purpose |
| --- | --- | --- |
| 4 Choices | Current classic answer buttons. | Preserve v1.0 default behavior. |
| True / False | Show a proposed answer; player taps True or False. | Fast variety with lower score because guessing chance is higher. |
| Complete | Player types the answer with a custom keypad. | Strongest educational signal, no guessing. |
| Shuffle | Randomly mixes formats per question. | Advanced variety for confident players. |

### Required Flutter Architecture

| Piece | Implementation direction |
| --- | --- |
| `QuestionFormat` enum | Add `choice4`, `trueFalse`, `complete`, `shuffle`. |
| Question model | Extend `Question` with format-specific fields such as proposed answer and truth value. |
| Runtime state | Store selected game format and current question format separately. |
| Config UI | Add Question Type selector using the same visual style as existing controls. |
| Answer widgets | Build separate `ChoiceAnswerGrid`, `TrueFalseAnswerGrid`, and `CompleteKeypad` widgets. |
| Format badge | Show compact current-format badge during Shuffle, Master, and Daily Boss. |
| Engine helpers | Centralize scoring multipliers, timer adjustments, answer validation, and power-up availability. |

### Scoring And Timing

| Format | Score multiplier | Timer adjustment |
| --- | --- | --- |
| True / False | 0.5x base points | -2 seconds in per-question timer modes. |
| 4 Choices | 1x base points | No change. |
| Complete | 2x base points | +5 seconds in per-question timer modes. |
| Shuffle | Per current question format. | Per current question format. |

Important: format multiplier should apply to base question reward, then existing mode multipliers and combo logic can apply afterward. This must be tested so the economy does not inflate by accident.

### Master And Daily Boss Strategy

| Mode | Format strategy |
| --- | --- |
| Normal games | Player-selected format from config. |
| Shuffle games | 50% 4 Choices, 25% True/False, 25% Complete. |
| Master Stage 1 | 4 Choices only. |
| Master Stage 2 | 4 Choices only. |
| Master Stage 3 | 75% 4 Choices, 25% True/False. |
| Master Stage 4 | 50% 4 Choices, 30% True/False, 20% Complete. |
| Master Stage 5 | 30% 4 Choices, 30% True/False, 40% Complete. |
| Daily Boss | Deterministic daily format, softened for retention: 50% 4 Choices, 30% True/False, 15% Complete, 5% Shuffle. |

The original uploaded plan suggested a harder Daily Boss mix. For Flutter v1.1, start softer because Daily Boss is a retention feature. We can increase difficulty later if testers like it.

### Complete Keypad

The keypad must be custom Flutter UI, not the native keyboard.

Required behavior:

- Digits 0-9.
- Decimal button enabled only for rationals.
- Minus button enabled only for integers and rationals.
- Backspace.
- Explicit Enter button.
- Empty input does nothing.
- Invalid partial input such as `-` or `.` does nothing.
- Rationals compare with epsilon tolerance.

### Format-Aware Power-Ups

| Power-up | 4 Choices | True / False | Complete |
| --- | --- | --- | --- |
| 50/50 | Works | Disabled | Disabled |
| Hint | Reveals correct choice | Disabled | Reveals first digit |
| Skip | Works | Works | Works |
| Time / Freeze | Works | Works | Works |
| Shield | Works where mode allows | Works where mode allows | Works where mode allows |

Disabled power-ups should stay visible but greyed out, with a clear toast or long-press explanation.

## Recommended v1.1 Scope

Ship v1.1 with a focused but powerful set:

1. Multi-Type Question System.
2. Mistake Review.
3. Boss Collection.
4. Format-aware high scores.
5. New format achievements.
6. First-launch v1.1 tutorial.
7. Celebration/confetti system if not already completed during clone parity.
8. Architecture extraction for question formats, celebrations, progression, and economy.

Do not include Weekly Boss Ladder, subscriptions, classroom plans, or remote analytics in v1.1. Those are better after we learn whether players actually use the new question formats.

## V1.1 Feature Details

### 1. Multi-Type Question System

This is the headline feature.

Expected player value:

- More variety.
- More educational challenge.
- Better fit for different skill levels.
- Strong foundation for future adaptive practice.

Risk:

- Complete mode may feel difficult or slow for younger kids.

Mitigation:

- Keep 4 Choices as default.
- Make Complete opt-in for normal games.
- Use soft Daily Boss weighting.
- Add Mistake Review so wrong answers feel educational, not punishing.

### 2. Mistake Review

After a game, show a review list of missed questions:

| Field | Example |
| --- | --- |
| Question | `7 × 8 = ?` |
| Your answer | `54` |
| Correct answer | `56` |
| Format | `Complete` |
| Skill | `Multiplication` |

Rules:

- Store missed questions only for the current game session.
- Do not persist child-level history in v1.1 unless Parent Report needs it.
- Add a button on results: `Review Mistakes`.
- If no mistakes, show a perfect-score celebration instead.

Why it belongs in v1.1:

- Complete mode is much more useful when kids see exactly what they missed.
- Parents can understand the educational value immediately.

### 3. Boss Collection

Every Daily Boss defeated should be saved into a collection book.

Suggested saved data:

| Field | Meaning |
| --- | --- |
| boss id/name | Which boss was defeated. |
| icon | Boss emoji. |
| date defeated | Local date key. |
| format | Today's boss format. |
| reward | Coins earned. |

UI:

- Add `Boss Book` or `Collection` button from Daily Boss modal or menu.
- Show defeated bosses as collectible cards.
- Locked bosses can appear as silhouettes or mystery cards.

Why it belongs in v1.1:

- It strengthens Daily Boss without adding a whole new mode.
- It works naturally with deterministic daily boss formats.

### 4. Format-Aware High Scores

High scores must store question format.

Required high-score fields:

- score
- mode
- operation
- difficulty
- number type
- question format
- player name/avatar
- date

Hall of Fame should either:

- show a filter by format, or
- show a small format badge beside every score.

This is important because Complete mode scoring is intentionally higher.

### 5. Daily Streak Calendar

Candidate for v1.1 if scope allows; otherwise v1.2.

Keep it simple:

- Show current streak.
- Show last 7 or 30 days.
- Mark days with completed Daily Boss or Daily Challenge.
- Give small daily chest reward.

Do not turn it into a large event system yet.

### 6. Smart Learning Mode

Candidate for v1.1 if Mistake Review is already done cleanly; otherwise v1.2.

Simple first version:

- Track accuracy by operation and number type locally.
- On results, recommend one practice type.
- Example: `Practice: Division with rationals`.

Do not call it AI. It can be a simple rules-based recommendation.

### 7. Avatar Closet Upgrade

Candidate for v1.1 only if clone parity and question formats are ahead of schedule.

Better structure:

- Owned bases.
- Owned hats.
- Owned accessories.
- Colors.
- Favorites.
- Recently used.

This is good Flutter UI work, but less important than learning/gameplay improvements.

### 8. Parent Report Screen

Candidate for v1.1 or v1.2.

Useful fields:

- Total questions answered.
- Overall accuracy.
- Accuracy by operation.
- Accuracy by question format.
- Strongest skill.
- Weakest skill.
- Best streak.
- Daily practice count.

Keep it local-only in v1.1.

### 9. Adaptive Practice 2.0

Better after v1.1 data exists.

Future behavior:

- If a child misses many Complete division questions, generate more division practice.
- If T/F is too easy, suggest Complete.
- If accuracy is low, reduce difficulty without making the child feel punished.

This should be v1.2 or v1.3, not v1.1 core.

### 10. Cleaner Architecture

This should happen gradually while building v1.1.

Target structure:

| Module | Responsibility |
| --- | --- |
| `GameEngine` | Mode rules, scoring, timers, answer flow. |
| `QuestionService` | Question generation and format selection. |
| `ProgressionService` | Achievements, daily challenges, skill progress. |
| `CelebrationService` | Confetti, emoji, sound, haptics. |
| `EconomyService` | Coins, shop, packs, local rewards. |
| `AdService` | AdMob loading and screen gating. |
| `PurchaseService` | IAP products, purchases, restore. |
| `AvatarService` | Avatar parts, owned items, custom saves. |

Do not rewrite everything at once. Extract the service that matches the current feature being implemented.

## Implementation Phases

### Phase 0: Clone Freeze

Goal: make sure v1.0 Flutter is stable before v1.1 work starts.

- Finish parity audit items.
- Fix release services.
- Build APK/AAB.
- Test core flows.
- Tag a stable v1.0 baseline.

### Phase 1: Question Format Foundation

Goal: add typed infrastructure without changing gameplay yet.

- Add `QuestionFormat`.
- Add selected format to config/state.
- Add current format to runtime state.
- Add format badge widget.
- Keep all formats rendering classic 4-choice temporarily.
- Add tests proving v1.0 4-choice path still works.

### Phase 2: True / False

Goal: T/F is playable end-to-end.

- Generate proposed answer from correct answer or plausible distractor.
- Add T/F answer widget.
- Add scoring multiplier.
- Add timer adjustment.
- Disable unsupported power-ups.
- Add T/F achievement tracking.

### Phase 3: Complete Keypad

Goal: typed-answer mode is playable end-to-end.

- Add custom keypad widget.
- Add natural/integer/rational validation.
- Add first-digit hint.
- Add scoring multiplier.
- Add timer adjustment.
- Add Complete achievements.
- Add strong tests for input edge cases.

### Phase 4: Shuffle, Master, Daily Boss

Goal: advanced mode integration.

- Add Shuffle weighted picker.
- Add Master stage format weights.
- Add Daily Boss deterministic format.
- Show format in Daily Boss modal.
- Add format-aware high scores.

### Phase 5: Mistake Review And Boss Collection

Goal: turn new formats into learning and retention features.

- Track missed questions during a session.
- Add Review Mistakes result action.
- Add Boss Collection data and UI.
- Add perfect-game celebration path when no mistakes.

### Phase 6: Tutorial, Polish, Closed Test

Goal: make v1.1 feel finished.

- Add one-time v1.1 tutorial.
- Polish phone/tablet layouts.
- Verify confetti/reduce-motion/low-performance behavior.
- Run closed testing with explicit format feedback questions.

## Testing Requirements

| Area | Required tests |
| --- | --- |
| 4-choice regression | Existing gameplay still behaves like v1.0. |
| T/F | True and false cases, plausible distractor, score, timer, achievements. |
| Complete | Empty input, negative input, decimals, epsilon comparison, score, timer. |
| Shuffle | Weighted distribution smoke test. |
| Master | Stage format weights and clear flow. |
| Daily Boss | Same date gives same boss format; retry gives fresh questions. |
| Power-ups | Unsupported power-ups disabled by format. |
| High scores | Same score table does not unfairly mix formats. |
| Mistake Review | Missed questions captured and displayed correctly. |
| Boss Collection | Boss defeat persists once with date and format. |

## What Not To Build In v1.1

- Subscriptions.
- Classroom plans.
- Remote analytics.
- Word problems.
- Drag-and-drop answers.
- Voice input.
- Custom question authoring.
- Weekly Boss Ladder unless v1.1 finishes far ahead of schedule.

## Success Criteria

v1.1 is successful if:

- Existing 4-choice users are not disrupted.
- At least one new format feels polished enough to become a favorite.
- Complete mode feels educational, not frustrating.
- Daily Boss is more interesting but not too hard.
- The codebase becomes cleaner, not heavier.
- The app still feels exactly like Math Challenge.
