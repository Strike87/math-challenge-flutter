## Goal

Implement v1.1 question types in a Flutter-native way:

- preserve the v1.0 clone behavior by default,
- keep 4-choice gameplay stable throughout development,
- add True/False, Complete, and Shuffle without turning `GameState` or `GameScreen` into another giant one-file app,
- prepare the code for Mistake Review, Boss Collection, Smart Learning, and Parent Report later.

## Current Code Anchors

| Area | Current file/class | Current role |
| --- | --- | --- |
| Question data | `lib\models\player.dart` / `Question` | Stores operation, key, text, answer, choices, difficulty, rational decimal places. |
| Enums | `lib\models\enums.dart` | Stores `Operation`, `Difficulty`, `NumberType`, `GameMode`, `PowerUp`. |

## Non-Negotiables

1. `QuestionFormat.choice4` must remain the default.
2. v1.0 gameplay must continue to work after Phase 1.
3. Existing question generation should not be rewritten during the first phase.
4. New UI must use existing Flutter theme/components.
5. Every new format must have unit tests before Master/Daily Boss integration.
6. Question format logic must be typed, not stringly typed.
7. Mistake Review should be designed into the answer pipeline from the start.

## New Types

### `QuestionFormat`

Add to `lib\models\enums.dart` or a new `lib\models\question_format.dart`.

Recommended enum:

```dart
enum QuestionFormat {
  choice4,
  trueFalse,
  complete,
  shuffle;

  String get label;
  String get shortLabel;
  double get scoreMultiplier;
  int get timerOffsetMs;
}
```

Recommended values:

| Format | Label | Badge | Score | Timer |
| --- | --- | --- | --- | --- |
| `choice4` | `4 Choices` | `4C` | `1.0` | `0` |
| `trueFalse` | `True / False` | `T/F` | `0.5` | `-2000` |
| `complete` | `Complete` | `✍️` | `2.0` | `+5000` |
| `shuffle` | `Shuffle` | `🎲` | dynamic | dynamic |

### `AnswerSubmission`

Do not make every format call `answer(num?)` forever. Add a typed submission object.

```dart
sealed class AnswerSubmission {
  const AnswerSubmission();
}

class ChoiceAnswerSubmission extends AnswerSubmission {
  final num value;
}

class TrueFalseSubmission extends AnswerSubmission {
  final bool value;
}

class CompleteAnswerSubmission extends AnswerSubmission {
  final String text;
}

class SkipSubmission extends AnswerSubmission {
  const SkipSubmission();
}
```

If sealed classes feel too heavy for the first pass, use a simple class:

```dart
class AnswerSubmission {
  final QuestionFormat format;
  final num? numericValue;
  final bool? boolValue;
  final String? textValue;
  final bool skipped;
}
```

Recommended path: start simple, then make it sealed when the engine is extracted.

### `QuestionAttempt`

Keep `Question` as the generated math payload. Add a wrapper for the current presentation format.

```dart
class QuestionAttempt {
  final Question question;
  final QuestionFormat format;
  final num? proposedAnswer;
  final bool? isTrueStatement;
  final List<num> visibleChoices;
  final DateTime startedAt;
}
```

Why this is better than stuffing everything into `Question`:

- the same math question can be presented in different formats,
- Shuffle/Master/Daily Boss can change presentation without changing the generator,
- Mistake Review can store both the math question and how it was shown.

First implementation shortcut:

- add fields directly to `Question` only if it keeps Phase 1 faster,
- but plan to move toward `QuestionAttempt` before Mistake Review.

### `MistakeRecord`

Add early even if UI ships later.

```dart
class MistakeRecord {
  final String questionText;
  final num correctAnswer;
  final String submittedAnswer;
  final Operation operation;
  final Difficulty difficulty;
  final NumberType numberType;
  final QuestionFormat format;
  final DateTime answeredAt;
}
```

Store only in memory for v1.1 unless Parent Report needs local aggregate stats.

### `BossCollectionEntry`

Use for defeated Daily Boss collection.

```dart
class BossCollectionEntry {
  final String bossName;
  final String icon;
  final String dateKey;
  final QuestionFormat format;
  final int reward;
}
```

Persist as JSON under `mc_boss_collection`.

## Engine Flow

### Current simplified flow

```text
_generateQ()
  -> QuestionGenerator.build()
  -> rt.q = Question(...)
GameScreen renders q.text and q.choices
answer(num?)
  -> compare val to q.ans
  -> score or wrong flow
  -> next question
```

### Target v1.1 flow

```text
_generateQ()
  -> QuestionGenerator.build() creates math payload
  -> QuestionFormatSelector resolves actual current format
  -> QuestionAttemptFactory decorates payload for selected format
  -> rt.attempt = QuestionAttempt(...)

GameScreen
  -> renders question text
  -> renders answer widget based on rt.attempt.format

submitAnswer(AnswerSubmission)
  -> AnswerEvaluator evaluates submission
  -> ScoreCalculator applies format and mode rules
  -> ProgressionService updates achievements/counters
  -> MistakeReview records wrong answer if needed
  -> next question or finish
```

### Compatibility wrapper

Keep the current 4-choice button path simple:

```dart
void answer(num? value) {
  submitAnswer(
    value == null
      ? const SkipSubmission()
      : ChoiceAnswerSubmission(value),
  );
}
```

This lets existing 4-choice UI keep working while new formats use `submitAnswer`.

## Format Selection Rules

### Normal modes

```text
if selectedFormat is choice4/trueFalse/complete:
  use selectedFormat for every question

if selectedFormat is shuffle:
  roll 50% choice4, 25% trueFalse, 25% complete per question
```

### Master mode

| Stage | Choice | T/F | Complete |
| --- | ---: | ---: | ---: |
| 1 | 100% | 0% | 0% |
| 2 | 100% | 0% | 0% |
| 3 | 75% | 25% | 0% |
| 4 | 50% | 30% | 20% |
| 5 | 30% | 30% | 40% |

### Daily Boss

Pick one deterministic format per local date.

Recommended v1.1 launch weights:

| Format | Weight |
| --- | ---: |
| 4 Choices | 50% |
| True / False | 30% |
| Complete | 15% |
| Shuffle | 5% |

If the selected Daily Boss format is Shuffle, use normal Shuffle weights per question.

## Answer Evaluation

### Choice 4

Correct if:

```dart
(submitted - correct).abs() < 1e-9
```

### True / False

Question attempt contains:

```dart
proposedAnswer
isTrueStatement
```

Correct if:

```dart
submission.value == attempt.isTrueStatement
```

Distractor rule:

- 50% use correct answer,
- 50% use one plausible distractor from existing choices,
- never use an absurd random distractor if a closer one exists.

### Complete

Validation rules:

| Number type | Accepted input |
| --- | --- |
| Natural | integer string only. No negative, no decimal. |
| Integers | optional leading minus, integer only. |
| Rationals | decimal allowed, optional leading minus. |

Ignore submit if:

- empty,
- only `-`,
- only `.`,
- only `-.`.

Correct if parsed value matches answer with epsilon.

## Scoring

Recommended scoring order:

1. Start with base question points.
2. Apply question format multiplier to base only.
3. Add fast/streak bonuses where original game does.
4. Apply mode/combo multipliers.
5. Add final rounded points.

This keeps Complete valuable without accidentally doubling every bonus.

Add tests for:

- T/F base points are half of classic,
- Complete base points are double classic,
- streak coins are not accidentally doubled,
- combo multiplier still applies after format multiplier.

## Timer Adjustments

Only apply format timer changes in per-question timer modes.

Do not apply per-question offsets to:

- Blitz global timer,
- Combo global timer,
- any future global-timer event mode.

For per-question modes:

```dart
duration = baseDuration + currentFormat.timerOffsetMs
duration = max(GameConfig.timerMinMs, duration)
```

Consider a higher minimum for Complete if testing shows the existing minimum is too strict.

## Power-Up Rules

Add helper:

```dart
bool isPowerUpAvailable(PowerUp pu, QuestionFormat format)
```

Rules:

| Power-up | Choice | T/F | Complete |
| --- | --- | --- | --- |
| 50/50 | yes | no | no |
| Hint / switchOp equivalent | yes | no | first digit reveal |
| Skip | yes | yes | yes |
| Time | yes | yes | yes |
| Shield | yes | yes | yes |
| Freeze | yes | yes | yes |

UI rule:

- disabled power-ups remain visible,
- disabled state uses greyed-out styling,
- tapping disabled power-up shows a short toast.

## Config Screen Changes

Add a Question Type selector after Game Mode and before Difficulty, unless the final UI looks better after Difficulty.

Default:

```dart
QuestionFormat.choice4
```

Options:

- 4 Choices,
- True / False,
- Complete,
- Shuffle.

Master and Daily Boss should ignore the config selection and show their own strategy in intro/modals.

## Mistake Review

Record wrong answers in `currentMistakes`.

For every wrong answer store:

- question text,
- correct answer,
- submitted answer as display string,
- format,
- operation,
- difficulty,
- number type.

Results modal behavior:

- if mistakes exist, show `Review Mistakes`,
- if no mistakes, show stronger perfect-score celebration,
- review modal shows concise rows and never feels like a punishment.

## Boss Collection

On Daily Boss win:

1. Save entry under `mc_boss_collection`.
2. Prevent duplicate entry for same boss/date.
3. Show collection button from Daily Boss modal or menu.

Keep UI simple in v1.1:

- unlocked cards show boss icon/name/date/format,
- locked cards can be mystery cards.

## Stop Conditions

Pause v1.1 implementation if:

- Phase 1 breaks existing 4-choice mode,
- Complete mode input feels clumsy after tablet/phone testing,
- Daily Boss becomes frustrating in closed testing,
- code churn grows too large before v1.0 clone is stable,
- any release-service bug appears in v1.0.
